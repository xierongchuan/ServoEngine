# План реализации: Оптимизация выходов из позиций в MACDX стратегии

## Цель
Реализовать точные и надёжные алгоритмы выхода из позиций для максимизации прибыли:
1. Выход на пике прибыли MACD тренда (когда импульс начинает ослабевать)
2. **Реалтайм-защита от пампов/дампов** через быстрый цикл мониторинга позиций (WebSocket)
3. Фиксация прибыли на сильных импульсных свечах
4. Экстренные выходы при развороте тренда

## Ключевое проектное решение: таймфрейм-агностичность

Текущий конфиг MACDX фиксирует `"timeframe": "1h"`, но стратегия должна корректно работать на **любом** таймфрейме (5m, 15m, 30m, 1h, 4h). Все параметры выражены в **свечах**, а не в абсолютных единицах времени. Абсолютное время рассчитывается динамически:

```python
# Утилита: перевод таймфрейма в минуты
TIMEFRAME_MINUTES = {"1m": 1, "5m": 5, "15m": 15, "30m": 30, "1h": 60, "4h": 240, "1D": 1440}

def tf_to_minutes(timeframe: str) -> int:
    return TIMEFRAME_MINUTES.get(timeframe, 60)

# Пример: max_hold_candles=80 на разных таймфреймах:
# 5m  → 80 × 5m  = 400m ≈ 6.7 часов
# 15m → 80 × 15m = 1200m = 20 часов
# 1h  → 80 × 1h  = 80h ≈ 3.3 дня
# 4h  → 80 × 4h  = 320h ≈ 13.3 дней
```

Все параметры в конфиге и в коде — **в свечах**. В логах и reason-строках выводим и свечи, и абсолютное время для наглядности.

## Анализ текущей логики выхода (MACDX)

### Существующие сигналы закрытия (`src/core/signals/macdx.py:should_close`):
- **MACD разворот + прибыль**: Гистограмма < 0 для лонгов + PnL >= 0.5% → закрытие (medium urgency)
- **MACD разворот + убыток**: Гистограмма < 0 для лонгов + PnL < -1.0% → закрытие (high urgency)
- **RSI экстремумы**: RSI > 80 для лонгов, RSI < 20 для шортов → безусловное закрытие
- **Прибыль с RSI**: PnL >= +3% при RSI > 70 (лонг) или RSI < 30 (шорт)
- **Зеркальная логика для шортов**: Аналогичные правила с hist > 0

### Сильные стороны текущей системы:
- Комбинированные условия (MACD + PnL), а не простые пороги
- Разная urgency для разных ситуаций
- Поддержка как dict так и object позиций

### Проблемы текущей системы:
- Выход только при полном развороте гистограммы (упущенная прибыль на ослаблении тренда)
- Нет отслеживания пиковой прибыли и просадки от пика
- Нет учёта времени в позиции (в свечах)
- Нет масштабирования порогов по волатильности (ATR)
- Порог -1% потерь статический, не учитывает плечо и ATR
- Нет интеграции с рыночным режимом (regime) при принятии решений о выходе
- **Нет реалтайм-мониторинга** между закрытиями свечей — пампы/дампы внутри свечи не детектируются
- `position_check_interval: 15` определён в `config/strategies/macdx.json`, но **не используется** в `run_macdx_loop()`

## Предлагаемая система выходов

### Двухуровневая архитектура (по аналогии с SCALP)

```
process_worker.py → run_macdx_loop()
  ├── Основной цикл (loop_interval: 60s) — анализ закрытых свечей, MACD, вход/выход
  └── NEW: Быстрый цикл мониторинга (position_check_interval: 15s) — только при открытой позиции
        ├── Читает текущую цену из WebSocket кэша (без REST запросов)
        ├── Обновляет peak_price, проверяет trailing stop
        ├── Детектирует пампы/дампы по % изменению цены
        └── Срочное закрытие при резком развороте
```

**Важно**: Быстрый цикл — это НЕ полный анализ. Он только:
1. Получает текущую цену из WebSocket кэша (`ws_provider.get_klines()`)
2. Сравнивает с entry_price и peak_price
3. Принимает решение close/hold

Это легковесная операция без REST-запросов, аналогичная `ScalpEngine._fast_loop()` в `src/core/scalp_engine.py:496-550`.

### 0. Быстрый цикл защиты позиций (NEW — ответ на памп/дамп)

#### Зачем нужен:
Даже на 1h таймфрейме памп/дамп может произойти **между закрытиями свечей** — за 5-30 секунд. Без быстрого мониторинга бот увидит результат только через 60 секунд (или до 60 минут — при закрытии свечи), когда прибыль уже потеряна или убыток вырос.

#### Алгоритм:
```python
def _position_guard_loop(self, symbol: str, position: dict, exit_context: dict, ws_provider):
    """
    Быстрый цикл мониторинга (каждые position_check_interval секунд).
    Работает ТОЛЬКО при открытой позиции.
    Не делает REST-запросов — только чтение из WebSocket кэша.
    """
    interval = self.preset.get("position_check_interval", 15)  # уже в macdx.json
    entry_price = float(position["entry_price"])
    pos_type = position.get("side", "BUY")
    leverage = self.preset.get("leverage", 6)
    
    while True:
        time.sleep(interval)
        
        # Получаем текущую цену из WS кэша (без REST)
        klines = ws_provider.get_klines(symbol, limit=1)
        if not klines:
            continue
        current_price = float(klines[-1]["close"])
        
        # Обновляем peak_price
        peak_price = exit_context.get("peak_price", entry_price)
        if pos_type == "BUY":
            peak_price = max(peak_price, current_price)
        else:
            peak_price = min(peak_price, current_price)
        exit_context["peak_price"] = peak_price
        
        # --- Детекция пампа/дампа ---
        # Процент изменения от peak_price (просадка от максимума)
        if pos_type == "BUY":
            drawdown_pct = (peak_price - current_price) / peak_price * 100
            pnl_pct = (current_price - entry_price) / entry_price * 100 * leverage
        else:
            drawdown_pct = (current_price - peak_price) / peak_price * 100
            pnl_pct = (entry_price - current_price) / entry_price * 100 * leverage

        # Обновляем peak_pnl
        peak_pnl = exit_context.get("peak_pnl", 0)
        peak_pnl = max(peak_pnl, pnl_pct)
        exit_context["peak_pnl"] = peak_pnl
        
        # 1. ЭКСТРЕННЫЙ СТОП: резкий разворот (дамп при лонге / памп при шорте)
        #    Если цена упала от пика > pump_reversal_pct — немедленное закрытие
        pump_reversal_pct = self.exit_rules.get("pump_guard", {}).get("reversal_pct", 1.5)
        if drawdown_pct >= pump_reversal_pct:
            return {"should_close": True, "reason": f"Pump reversal: -{drawdown_pct:.2f}% from peak", "urgency": "critical"}
        
        # 2. TRAILING STOP (быстрый): ATR trailing от peak_price
        atr = exit_context.get("last_atr", 0)
        if atr > 0:
            atr_mult = self.exit_rules.get("trailing_stop", {}).get("atr_multiplier", 1.5)
            stop_distance = atr * atr_mult
            if pos_type == "BUY" and current_price <= peak_price - stop_distance:
                return {"should_close": True, "reason": f"Fast ATR trailing stop (peak={peak_price:.2f})", "urgency": "high"}
            if pos_type == "SELL" and current_price >= peak_price + stop_distance:
                return {"should_close": True, "reason": f"Fast ATR trailing stop (peak={peak_price:.2f})", "urgency": "high"}
        
        # 3. ФИКСАЦИЯ ПРИБЫЛИ на пампе в нашу сторону
        #    Если PnL вырос > pump_profit_pct за время позиции — зафиксировать
        pump_profit_pct = self.exit_rules.get("pump_guard", {}).get("profit_lock_pct", 5.0)
        if pnl_pct >= pump_profit_pct:
            return {"should_close": True, "reason": f"Pump profit lock: +{pnl_pct:.1f}%", "urgency": "high"}
        
        # 4. МАКСИМАЛЬНЫЙ УБЫТОК (быстрая проверка)
        max_loss_pct = self.exit_rules.get("emergency", {}).get("max_loss_pct", 3.0)
        if pnl_pct <= -max_loss_pct:
            return {"should_close": True, "reason": f"Max loss guard: {pnl_pct:.1f}%", "urgency": "critical"}
    
    return {"should_close": False}
```

#### Интеграция в run_macdx_loop:
```python
def run_macdx_loop(symbol, ...):
    """Основной цикл MACDX с быстрым мониторингом позиций."""
    while True:
        # ... существующая логика: analyze → generate/should_close → execute ...
        
        # После основного цикла — быстрый мониторинг если есть позиция
        position = trade_tracker.get_position(symbol)
        if position and ws_provider:
            # Быстрая проверка каждые position_check_interval секунд
            # до следующего основного цикла (loop_interval)
            remaining_time = loop_interval  # 60 секунд
            check_interval = preset.get("position_check_interval", 15)
            
            while remaining_time > 0:
                time.sleep(min(check_interval, remaining_time))
                remaining_time -= check_interval
                
                guard_result = position_guard_check(symbol, position, exit_context, ws_provider, exit_rules)
                if guard_result.get("should_close"):
                    executor.close_position(symbol, guard_result["reason"])
                    break
        else:
            time.sleep(loop_interval)
```

**Ключевое отличие от SCALP**: SCALP использует полноценный dual-loop с threading. Для MACDX мы НЕ создаём отдельный поток — быстрый мониторинг работает внутри основного цикла, заменяя `time.sleep(60)` на серию коротких проверок. Это проще и надёжнее.

### 1. Выход на ослаблении импульса MACD (пиковая прибыль)

#### Алгоритм отслеживания пика гистограммы:
```python
class MacdMomentumTracker:
    """Отслеживает пик гистограммы MACD для определения ослабления тренда.
    
    Работает на уровне свечей (таймфрейм из конфига), не в реальном времени.
    Обновляется при каждом основном цикле (каждые loop_interval секунд).
    """
    def __init__(self):
        self.peak_hist = 0           # Абсолютный пик гистограммы с момента входа
        self.peak_candle_idx = 0     # Индекс свечи с пиком
        self.candles_since_entry = 0 # Количество свечей с момента входа

    def update(self, hist: float):
        """Обновить при каждом новом значении гистограммы."""
        self.candles_since_entry += 1
        abs_hist = abs(hist)
        if abs_hist > abs(self.peak_hist):
            self.peak_hist = hist
            self.peak_candle_idx = self.candles_since_entry

    def get_weakening_ratio(self, current_hist: float) -> float:
        """Отношение текущей гистограммы к пику. 1.0 = на пике, 0.0 = полный разворот."""
        if self.peak_hist == 0:
            return 1.0
        # Для лонгов: peak_hist > 0, ratio падает при уменьшении hist
        # Для шортов: peak_hist < 0, ratio падает при увеличении hist (к 0)
        if self.peak_hist > 0:
            return max(0, current_hist / self.peak_hist)
        else:
            return max(0, current_hist / self.peak_hist) if current_hist < 0 else 0

    def candles_since_peak(self) -> int:
        """Количество свечей после пика."""
        return self.candles_since_entry - self.peak_candle_idx
```

#### Динамические уровни выхода на основе ослабления:
```python
# Контекст: обновление каждый основной цикл, все пороги — в свечах
tracker = position_context.get("macd_tracker")
weakening = tracker.get_weakening_ratio(macd_hist)
candles_after_peak = tracker.candles_since_peak()

# Уровень 1: Консервативный выход (50% от пика + 3+ свечей после пика)
# Пример: hist пик=0.8, текущий=0.4 → weakening=0.5
# На 5m: 3 свечи = 15 минут. На 1h: 3 свечи = 3 часа. На 4h: 3 свечи = 12 часов.
if weakening <= 0.5 and candles_after_peak >= 3 and pnl_pct >= 0.5:
    return {"should_close": True, "reason": f"MACD momentum weakened to {weakening:.0%} (peak exit)", "urgency": "medium"}

# Уровень 2: Пересечение сигнальной линии (запасной сигнал)
if position_long and macd_line < macd_signal and prev_macd_line >= prev_macd_signal:
    return {"should_close": True, "reason": "MACD bearish signal crossover", "urgency": "medium"}
```

**Важно**: Ослабление до 70% слишком агрессивно — нормальные откаты тренда легко достигают 60-70% от пика на любом таймфрейме. Порог 50% + минимум 3 свечи после пика — баланс между фиксацией прибыли и удержанием тренда.

### 2. Фиксация прибыли на импульсных свечах

```python
# Импульсная свеча: тело > 2 * ATR (аномально большая свеча для ЛЮБОГО таймфрейма)
# ATR автоматически масштабирован под таймфрейм (analyzer считает его по текущим свечам)
candle_body = abs(close - open_price)
is_impulse_candle = candle_body > atr * 2.0

# Фиксация на импульсной свече с прибылью
if is_impulse_candle and pnl_pct >= min_profit_for_impulse:  # порог из regime adaptation
    return {"should_close": True, "reason": f"Impulse candle +{pnl_pct:.1f}% (body={candle_body/atr:.1f}x ATR)", "urgency": "high"}

# Объёмное подтверждение импульса
if is_impulse_candle and volume_ratio > 2.5 and pnl_pct >= 1.0:
    return {"should_close": True, "reason": f"Volume impulse +{pnl_pct:.1f}% (vol={volume_ratio:.1f}x)", "urgency": "high"}
```

**Примечание**: На коротких таймфреймах (5m, 15m) импульсные свечи встречаются чаще, но ATR масштабирован соответственно — условие `body > 2x ATR` остаётся корректным фильтром для аномальных движений на любом таймфрейме.

### 3. Trailing profit (защита накопленной прибыли)

```python
# Отслеживание пика прибыли (обновляется в обоих циклах — основном и быстром)
peak_pnl = position_context.get("peak_pnl", 0)
peak_pnl = max(peak_pnl, pnl_pct)
position_context["peak_pnl"] = peak_pnl

# Trailing profit: закрываем если прибыль упала более чем на N% от пика
# Пример: пик +5%, текущая +3% → просадка от пика 40% → закрытие
if peak_pnl >= 2.0:  # Активируется только при значительной прибыли
    drawdown_from_peak = (peak_pnl - pnl_pct) / peak_pnl
    
    # Адаптивный порог: чем больше прибыль, тем строже защита
    if peak_pnl >= 5.0:
        max_drawdown = 0.30  # Защищаем 70% прибыли при пике 5%+
    elif peak_pnl >= 3.0:
        max_drawdown = 0.40  # Защищаем 60% прибыли при пике 3-5%
    else:
        max_drawdown = 0.50  # Защищаем 50% прибыли при пике 2-3%
    
    if drawdown_from_peak >= max_drawdown:
        return {
            "should_close": True,
            "reason": f"Trailing profit: peak +{peak_pnl:.1f}%, now +{pnl_pct:.1f}% (drawdown {drawdown_from_peak:.0%})",
            "urgency": "medium"
        }
```

### 4. Защитные и адаптивные механизмы

#### ATR-адаптивный стоп-лосс (trailing):
```python
# ATR-based trailing stop — привязан к пиковой цене, не к entry
# ATR автоматически масштабирован под таймфрейм (рассчитан analyzer'ом)
atr_mult = config.get("preset", {}).get("atr_sl_mult", 1.5)
stop_distance = atr * atr_mult  # 1.5 * ATR по умолчанию

# Активируется при прибыли >= 1 ATR
activation_threshold = atr
if pos_type == "BUY":
    if current_price > entry_price + activation_threshold:
        # Trailing от максимальной цены
        peak_price = position_context.get("peak_price", current_price)
        peak_price = max(peak_price, current_price)
        position_context["peak_price"] = peak_price
        trailing_stop = peak_price - stop_distance
        
        if current_price <= trailing_stop:
            return {"should_close": True, "reason": f"ATR trailing stop hit ({trailing_stop:.2f})", "urgency": "high"}
```

#### Экстренные выходы при развороте тренда:
```python
def should_emergency_close(position, analysis, macd_hist, macd_hist_prev):
    """Экстренные выходы — только при подтверждённых разворотах."""
    pos_type = "BUY" if position.is_long else "SELL"
    
    # Защита: не закрывать если позиция соответствует текущему MACD тренду
    position_matches_trend = (
        (pos_type == "BUY" and macd_hist > 0) or
        (pos_type == "SELL" and macd_hist < 0)
    )
    if position_matches_trend:
        return {"should_close": False, "reason": "Position matches MACD trend"}
    
    # 1. Резкий разворот гистограммы (смена знака + значительная величина)
    atr = analysis.get("atr", 0)
    macd_reversal_threshold = atr * 0.001  # Масштабирован к ATR
    
    if pos_type == "BUY" and macd_hist_prev > 0 and macd_hist < -macd_reversal_threshold:
        return {"should_close": True, "reason": "MACD instant reversal (bearish)", "urgency": "critical"}
    if pos_type == "SELL" and macd_hist_prev < 0 and macd_hist > macd_reversal_threshold:
        return {"should_close": True, "reason": "MACD instant reversal (bullish)", "urgency": "critical"}
    
    # 2. Убыток превышает 2 * ATR (динамический макс. убыток)
    entry_price = float(position.entry_price)
    current_price = analysis.get("current_price", 0)
    if pos_type == "BUY":
        loss_in_atr = (entry_price - current_price) / atr if atr > 0 else 0
    else:
        loss_in_atr = (current_price - entry_price) / atr if atr > 0 else 0
    
    if loss_in_atr > 2.0:
        return {"should_close": True, "reason": f"Max loss exceeded ({loss_in_atr:.1f}x ATR)", "urgency": "critical"}
    
    return {"should_close": False, "reason": "No emergency conditions"}
```

#### Учёт рыночного режима при выходе:
```python
# Режим уже рассчитывается в MacdxPipeline.generate_command()
# Используем его для адаптации порогов выхода

regime = analysis.get("regime", {}).get("regime", "UNKNOWN")

if regime == "TRENDING":
    # В тренде — даём позиции больше пространства
    weakening_threshold = 0.40    # Закрываем при 40% от пика (агрессивное удержание)
    trailing_drawdown = 0.50      # Допускаем 50% просадки от пика прибыли
    min_profit_for_impulse = 2.0  # Импульсные выходы только при 2%+ прибыли
elif regime == "RANGING":
    # В боковике — фиксируем быстрее
    weakening_threshold = 0.60    # Закрываем при 60% от пика (быстрая фиксация)
    trailing_drawdown = 0.35      # Допускаем только 35% просадки
    min_profit_for_impulse = 0.8  # Импульсные выходы уже при 0.8%
elif regime == "VOLATILE":
    # В волатильности — защита прежде всего
    weakening_threshold = 0.55
    trailing_drawdown = 0.30
    min_profit_for_impulse = 1.0
else:  # TRANSITIONAL / UNKNOWN
    weakening_threshold = 0.50
    trailing_drawdown = 0.40
    min_profit_for_impulse = 1.5
```

#### Анти-паника фильтры (в свечах, таймфрейм-агностично):
- **Минимальное количество свечей**: Не закрывать по ослаблению раньше 2-х полных свечей после входа
  - На 5m: 10 минут. На 1h: 2 часа. На 4h: 8 часов — пропорционально таймфрейму
- **Подтверждение тренда**: Требовать что ослабление продолжается 2+ свечи подряд (не просто откат одной свечи)
- **Не закрывать в плюсе при сильном ADX**: Если ADX > 30 и позиция прибыльна — удерживать (сильный тренд)
- **Быстрый цикл НЕ анализирует MACD**: Он только следит за ценой — решения по индикаторам принимает основной цикл

#### Риск-менеджмент:
- **Максимальная просадка**: ATR-масштабированная (2x ATR), не фиксированные проценты
- **Временной лимит**: `max_hold_candles` свечей (по умолчанию 80) — абсолютное время зависит от таймфрейма
- **Максимальный стоп**: Плечо × ATR × SL_mult — учитывает конфигурацию левеража (6x)
- **Быстрый guard**: `position_check_interval` секунд (по умолчанию 15) — одинаков для всех таймфреймов, т.к. пампы не зависят от выбранного таймфрейма

## Техническая реализация

### Расширение should_close с приоритетами (основной цикл):
```python
def should_close(self, analysis: Dict, position: Any, **kwargs) -> Dict:
    """
    Приоритизация выходов (основной цикл — по закрытым свечам):
    1. Экстренные выходы (critical) — немедленное закрытие
    2. Trailing profit / ATR trailing stop (high) — защита прибыли
    3. Ослабление MACD импульса (medium) — оптимальный выход на пике
    4. Импульсные свечи + прибыль (medium) — фиксация на волатильности
    5. Текущие правила (RSI экстремумы, MACD разворот + PnL)
    
    Примечание: пампы/дампы между свечами обрабатываются
    быстрым циклом (position_guard) в process_worker.
    """
    if not position:
        return {"should_close": False, "reason": "No position", "urgency": "low"}

    # ... existing position parsing ...
    
    position_context = kwargs.get("position_context", {})
    regime = analysis.get("regime", {})

    # 1. ЭКСТРЕННЫЕ ВЫХОДЫ — максимальный приоритет
    emergency = self._check_emergency_exit(analysis, position, pos_type, pnl_pct)
    if emergency.get("should_close"):
        return emergency

    # 2. TRAILING PROFIT — защита накопленной прибыли
    trailing = self._check_trailing_profit(pnl_pct, position_context, regime)
    if trailing.get("should_close"):
        return trailing

    # 3. ОСЛАБЛЕНИЕ MACD ИМПУЛЬСА — выход на пике тренда
    momentum = self._check_macd_weakening(analysis, position_context, pnl_pct, regime)
    if momentum.get("should_close"):
        return momentum

    # 4. ИМПУЛЬСНЫЕ СВЕЧИ — фиксация на волатильности
    impulse = self._check_impulse_exit(analysis, pnl_pct, regime)
    if impulse.get("should_close"):
        return impulse

    # 5. СУЩЕСТВУЮЩИЕ ПРАВИЛА (MACD reversal + PnL, RSI extreme, TP)
    return self._check_standard_exits(analysis, pos_type, pnl_pct)
```

## Технические вызовы и решения

### Сохранение exit_context между циклами:
- **Проблема**: MacdxSignalGenerator создаётся заново каждый цикл (`gen = MacdxSignalGenerator(settings)`)
- **Решение**: Хранить exit_context в active_trades.json через TradeTracker. Передавать через kwargs в should_close
- **Альтернатива**: Сделать MacdxSignalGenerator singleton per symbol (как MarketRegimeDetector)

### Доступ к WebSocket из быстрого цикла:
- **Проблема**: WebSocket provider — multiprocessing Manager proxy, доступ может быть медленным
- **Решение**: `get_klines(symbol, limit=1)` — минимальный запрос, ~1мс через proxy. Если WebSocket недоступен — пропускаем быструю проверку, основной цикл продолжает работать
- **Fallback**: Если `ws_provider is None`, быстрый цикл отключается, `time.sleep(loop_interval)` как раньше

### Совместимость с бэктестом:
- **Проблема**: Бэктест использует тот же MacdxSignalGenerator — новая логика должна работать и в бэктесте
- **Решение**: exit_context передаётся через kwargs, бэктест может создавать свой context dict
- **Быстрый цикл**: В бэктесте не нужен — пампы симулируются через данные свечей

### Обратная совместимость:
- **Проблема**: Существующие позиции не имеют exit_context
- **Решение**: Все новые проверки имеют `if not position_context: return False` — если контекста нет, работают только старые правила

### Таймфрейм-агностичность:
- **Проблема**: Параметры должны работать одинаково хорошо на 5m и 4h
- **Решение**: Все пороги времени — в свечах. ATR автоматически масштабирован. Проценты — таймфрейм-агностичны. Логи выводят и свечи и абсолютное время: `f"{candles} candles ({candles * tf_minutes}m)"`

### Надёжность WebSocket для быстрого цикла (анализ `ws_data_provider.py`):

Быстрый цикл (каждые 15 секунд) читает текущую цену из WebSocket кэша. Ниже — анализ надёжности и необходимые меры.

#### Текущие механизмы защиты (уже реализованы):
- **REST backfill** при старте — загружает 600 свечей для каждого символа
- **Auto-reconnect** с exponential backoff (1с → 60с макс.)
- **Keepalive ping** каждые 20 секунд — предотвращает idle disconnect
- **REST подтверждение завершённых свечей** — при смене периода запрашивает окончательные данные из REST API
- **is_cache_ready()** — проверка наличия достаточных данных перед использованием
- **Manager dict proxy** — потокобезопасные операции через IPC

#### Частота обновления WebSocket кэша:
- BingX WebSocket отправляет обновления kline **каждые ~1-2 секунды** (реалтайм тик-данные)
- Каждое обновление обновляет текущую (незакрытую) свечу в кэше через `_update_cache()`
- При 15-секундном polling быстрого цикла, кэш будет обновлён **~10-15 раз** между проверками
- Это более чем достаточно для детекции пампов/дампов — задержка ≤ 2 секунды

#### Производительность IPC (Manager dict proxy):
- Каждый `get_klines(symbol, limit=1)` — один RPC через Unix socket к Manager процессу
- Латентность: ~1-5 мс на вызов (локальный IPC)
- При 15-секундном интервале и 2-5 символах: ~0.1-0.3 запроса/сек — **пренебрежимо мало**
- Для сравнения: SCALP `_fast_loop()` читает каждые 1.5 секунды и работает стабильно

#### Выявленные режимы отказа и меры:

**1. WebSocket отключение → кэш перестаёт обновляться:**
- **Текущее**: auto-reconnect с backoff, но **нет детекции устаревших данных** в worker
- **Решение для плана**: Добавить проверку `last_update_timestamp` в быстром цикле:
```python
# В position_guard_check():
klines = ws_provider.get_klines(symbol, limit=1)
if not klines:
    info(f"[GUARD] {symbol}: No WS data, skipping guard check")
    continue

# Проверка свежести данных
candle_age = time.time() - klines[-1]["timestamp"] / 1000
max_staleness = 60  # Максимум 60 секунд устаревания
if candle_age > max_staleness:
    warning(f"[GUARD] {symbol}: WS data stale ({candle_age:.0f}s old), skipping guard")
    continue
```

**2. REST backfill не удалось → символ остаётся `is_ready=False`:**
- **Текущее**: нет retry — символ навсегда в состоянии "не готов"
- **Решение для плана**: Быстрый цикл НЕ зависит от `is_ready` — он читает `get_klines()` напрямую. Если данных нет — пропускает проверку (graceful degradation). Дополнительно: добавить периодический re-backfill (каждые 5 минут) для символов с `is_ready=False`

**3. Частичная запись (race condition read-modify-write):**
- **Текущее**: WS thread — единственный writer. Workers — read-only. Manager dict атомарен на уровне операций
- **Риск**: При одновременном чтении worker и записи WS thread — worker может получить промежуточное состояние (свеча обновлена наполовину)
- **Реальный impact**: Минимальный — свеча отличается на 1-2 тика, что для порогов 1.5% несущественно
- **Решение для плана**: Использовать `list()` копию (уже делается в `get_klines()` через `list(self._kline_cache.get(...))`), что создаёт snapshot

**4. Manager процесс упал → все workers теряют доступ к кэшу:**
- **Текущее**: нет recovery
- **Решение для плана**: Быстрый цикл с try/except — при ошибке IPC логирует warning и продолжает без guard. Основной цикл с REST API продолжает работать

- **Митигация**: Быстрый цикл может только ЗАКРЫТЬ позицию, не открыть. Основной цикл обнаружит закрытие и обновит состояние
- **Митигация**: exit_context общий для обоих циклов (peak_price, peak_pnl обновляются в обоих)
- **Митигация**: Быстрый цикл работает внутри основного (не в отдельном потоке), поэтому нет race conditions

- **Митигация**: `pump_reversal_pct = 1.5%` — достаточно высокий порог, обычный шум < 0.5%
- **Митигация**: Проверка только от peak_price, а не от entry — не срабатывает на начальном движении

- **Митигация**: Все параметры в конфиге, не в коде
